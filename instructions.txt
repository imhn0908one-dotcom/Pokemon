# 🤖 Windsurf 開発指示書: パーティ管理設計の刷新とダメージ計算器の土台構築

## 1. 現状 (Current Status)

- `PartyBuilder` クラス（`party_builder.py`）および `PartyBuilderCUI`（`partybuilder_CUI.py`）が実装され、CUIとロジックの分離、DI（依存性注入）の仕組みが整っている。
- しかし、現在は「対話入力がすべて完了した最後に一括でTOMLへ書き出す」仕様になっている。これでは、入力途中でのクラッシュや中断に対応できず、今後「ダメージ計算器」の中でステータスをその場でポチポチ書き換えて何度も再計算するような、リアルタイムな編集・テスト要件を満たせない。

## 2. 目的 (Objective)
「新規作成」と「途中編集」という別々のフローを完全に撤廃し、「最初に空（デフォルト値）のパーティTOMLを即時生成し、以降はすべて『指定枠のデータをチェンジして即保存する』という単一のデータフローに一本化」する。

これにより、堅牢でいつでも中断可能であり、将来のGUI化やダメージ計算器との連携が極めて容易なパーティ管理ロジック `PartyManagerLogic` を構築する。

## 3. 綺麗にするべきポイント (Refactoring Points)

- **`PartyBuilder` から `PartyManagerLogic` への昇格と集約:**

- 「一から組み立てる」ロジックを廃止し、「枠（0〜5）を指定して中身を上書き（チェンジ）する」関数群に統一する。
- **ファイル・メモリの常時同期（即時セーブ）:**

- 努力値、技、性格、ポケモン種族などが「チェンジ」されたら、それぞれの関数内部で自動的に `save_changes()`（TOMLへの物理上書き保存）を呼び出す。
- **データ構造の確定（IDと表示名の併記）:**

- TOMLの可読性とエディタ（UI層）での表示を助けるため、`pokemon_id` や `nature_id` と共に、DBから引っ張ってきた `name` や `nature_name` もデータに含めて保持・保存する。
- 実数値（HPや攻撃力など）はTOMLには保存せず、`PokemonInstance` が読み込む際に努力値や性格から都度自動計算する（単一光源の原則）。

## 4. 望まれる結果 (Expected Outcomes)

### ① 新しいTOMLデータ構造のサポート
新規作成（`create_empty_party`）を実行した瞬間、以下のような「未選択（デフォルト値）」の6匹分のテーブル配列が含まれるTOMLが物理生成されること。

```toml
party_name = "（入力されたパーティ名）"

[[pokemon]]
pokemon_id = 0
name = "（未選択）"
level = 50
nature_id = 1
nature_name = "Hardy"
ability_id = 0
ability_name = "（未選択）"
moves = [0, 0, 0, 0]
[pokemon.evs]
hp = 0
attack = 0
defense = 0
special_attack = 0
special_defense = 0
speed = 0
#これが6匹分繰り返される
```

### ② ロジック層 (`party_manager_logic.py`) の実装
画面入出力（`print`/`input`）を一切持たない純粋な関数群として以下を実装する。

- `create_empty_party(party_name)`: 上記の空テンプレを即時保存。
- `load_party(filepath)`: 既存TOMLのロード。
- `change_pokemon_species(index, pokemon_id)`: ポケモンの種類をチェンジ（技やEVは安全のため0にリセット、DBから名前を自動取得してセット、即保存）。
- `change_pokemon_ev(index, stat_name, value)`: 努力値をチェンジ（即保存）。
- `change_pokemon_move(index, slot_index, move_id)`: 技をチェンジ（即保存）。
- `save_changes()`: TOMLファイルへの上書き物理保存。

### ③ CUI層 (`party_manager_cui.py`) の実装
新規・編集を区別せず、常に「現在の6匹のリストを表示 ➔ スロット（0〜5）を選択 ➔ 何をチェンジするか選択（ポケモン変更/技変更/努力値調整）」という**共通のループ構造**にリファクタリングする。
